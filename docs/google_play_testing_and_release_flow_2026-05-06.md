# 구글 플레이 테스트 및 출시 흐름 정리 (내차수첩)

기준일: 2026-05-06  
앱명: `내차수첩`  
패키지명: `com.kogit.mycarnote`

이 문서는 Google Play Console에서 앱을 정식 출시하기 전 진행하는 테스트 절차와, 테스트 이후 프로덕션 출시까지의 흐름을 정리한 문서입니다.  
특히 `내부 테스트`, `비공개 테스트`, `공개 테스트`, `프로덕션 출시`의 차이와 실제 운영 순서를 중심으로 정리합니다.

공식 문서 기준으로 작성했으며, 현재 프로젝트 기준 실행 방식도 함께 반영했습니다.

---

## 1. 전체 흐름

Google Play 앱 출시는 보통 아래 순서로 진행합니다.

1. Play Console에서 앱 생성
2. 스토어 등록 정보 입력
3. 앱 콘텐츠(App content) 입력
4. 내부 테스트(Internal testing)
5. 비공개 테스트(Closed testing)
6. 필요 시 공개 테스트(Open testing)
7. 프로덕션(Production) 출시

실무적으로는 아래 흐름을 가장 많이 사용합니다.

1. 내부 테스트
2. 비공개 테스트
3. 프로덕션 출시

즉, `공개 테스트`는 항상 필수는 아닙니다.  
다만 앱 상태가 불안정하거나 외부 공개 베타를 운영하려면 공개 테스트를 거칠 수 있습니다.

---

## 2. 테스트 트랙 종류

Google Play에는 출시 전 사용할 수 있는 테스트 트랙이 3가지 있습니다.

### 2-1. 내부 테스트 (Internal testing)

가장 빠르게 앱을 배포하는 테스트 방식입니다.

특징:

- 최대 `100명`까지 테스트 가능
- 이메일 리스트 기반으로 초대
- 가장 빠르게 배포 반영
- Play Store 검색에는 보이지 않음
- 공유 링크를 받아야 설치 가능

주 용도:

- 개발 직후 QA
- 치명 버그 확인
- 기본 설치/로그인/기능 작동 확인
- 출시 직전 빠른 재점검

장점:

- 배포 속도가 빠름
- 설정이 단순함
- 앱 초기 상태에서도 사용 가능

한계:

- 소수 인원 테스트에 적합
- 실제 대규모 사용자 검증에는 부족

---

### 2-2. 비공개 테스트 (Closed testing)

특정 사용자 그룹을 대상으로 앱을 더 넓게 검증하는 단계입니다.

특징:

- 특정 이메일 그룹 또는 Google Group 기반
- 외부 일반 사용자에게 공개되지 않음
- 여러 개의 비공개 트랙 운영 가능
- 실제 출시 전 가장 중요한 테스트 단계

주 용도:

- 실사용 검증
- 여러 기기에서 안정성 확인
- 저장/불러오기/동기화 로직 확인
- UI/UX 피드백 수집
- 광고, 정책 문서, 계정 흐름 검증

장점:

- 공개 없이 실제 사용자 반응을 받을 수 있음
- 내부 테스트보다 현실적인 피드백 확보 가능
- 개인 개발자 계정의 테스트 요건 대응에 유리

한계:

- 내부 테스트보다 운영 관리가 조금 더 필요함
- 테스터 그룹 관리가 필요함

---

### 2-3. 공개 테스트 (Open testing)

앱의 테스트 버전을 Google Play 사용자에게 공개하는 베타 단계입니다.

특징:

- 누구나 참여 가능하게 열 수 있음
- Play에서 테스트 앱이 노출될 수 있음
- 대규모 테스트에 적합

주 용도:

- 출시 직전 공개 베타 운영
- 많은 사용자로부터 피드백 수집
- 대규모 기기 환경 테스트

장점:

- 실제 운영에 가까운 피드백 확보
- 많은 사용자 검증 가능

한계:

- 스토어 등록 정보와 정책 문서가 거의 완성되어 있어야 함
- 앱이 외부에 노출되므로 미완성 상태에는 부적합

---

## 3. 테스트 트랙 차이 요약

### 내부 테스트

- 빠른 QA
- 최대 100명
- 비공개 링크 중심
- 개발/초기 점검용

### 비공개 테스트

- 실제 사용 검증
- 신뢰 가능한 사용자 그룹 대상
- 출시 전 핵심 검증 단계

### 공개 테스트

- 공개 베타
- 넓은 사용자 대상
- 거의 출시 직전 단계

## 4. 현재 앱 기준 추천 순서

`내차수첩` 현재 상태 기준으로는 아래 순서를 추천합니다.

1. 내부 테스트
2. 비공개 테스트
3. 문제 없으면 프로덕션 출시

공개 테스트는 아래 경우에만 고려하면 됩니다.

- 외부 사용자 수를 더 넓혀보고 싶을 때
- 출시 전 대규모 환경 검증이 필요할 때
- 마케팅 목적의 베타 운영을 하고 싶을 때

즉, 현재 앱은 `공개 테스트 없이도` 내부/비공개 테스트만으로 출시 가능성이 있습니다.

---

## 5. 내부 테스트 진행 방법

경로:

- `Play Console > Test and release > Testing > Internal testing`

진행 순서:

1. Internal testing 페이지 진입
2. `Testers` 탭에서 이메일 리스트 생성
3. 내부 테스터 이메일 추가
4. `Create new release`
5. 최신 `AAB` 업로드
6. 릴리스 노트 입력
7. 검토 후 배포
8. 생성된 테스트 링크를 테스터에게 전달

현재 업로드 파일:

- `web/android/app/build/outputs/bundle/release/app-release.aab`

내부 테스트에서 확인할 항목:

- 설치/업데이트 가능 여부
- 로그인 가능 여부
- 비회원 시작 가능 여부
- 차량 등록/삭제
- 대시보드 로딩
- 주행거리 업데이트
- 오일/필터/소모품 기록
- 주유/충전 기록
- 타이어 기록
- 광고 표시
- 설정/정책 문서 표시

실무 팁:

- 내부 테스트는 소수 인원으로 짧게 진행
- 치명 버그가 사라지면 빠르게 비공개 테스트로 이동

---

## 6. 비공개 테스트 진행 방법

경로:

- `Play Console > Test and release > Testing > Closed testing`

진행 순서:

1. Closed testing 페이지 진입
2. 기본 트랙 사용 또는 새 closed track 생성
3. 테스터 그룹 설정
4. `Create new release`
5. 최신 `AAB` 업로드
6. 릴리스 노트 입력
7. 배포
8. 초대 링크 또는 대상 그룹으로 배포

추천 테스터 유형:

- 실제 차량 사용자
- 가족 차량/업무 차량 여러 대를 쓰는 사용자
- 장거리 운전자
- 기록 기능을 꾸준히 사용할 수 있는 사용자
- 서로 다른 제조사/차종/연료 타입 사용자

비공개 테스트에서 집중 확인할 것:

- 앱이 며칠 이상 써도 안정적인지
- 차량 데이터 저장/불러오기 일관성
- 대시보드 재진입 속도
- 날짜 기반/거리 기반 점검 항목 계산
- 모달, 저장 메시지, 스크롤 동작
- 광고 위치와 방해 요소 여부
- 계정/비회원 재접속 흐름

실무 팁:

- 내부 테스트보다 더 긴 기간 운영
- 최소 1회 이상 실제 주행/정비/기록 흐름을 끝까지 써보게 하는 것이 좋음

---

## 7. 공개 테스트 진행 방법

경로:

- `Play Console > Test and release > Testing > Open testing`

진행 순서:

1. Open testing 페이지 진입
2. `Testers` 설정
3. 피드백용 이메일 또는 URL 입력
4. `Create new release`
5. AAB 업로드
6. 릴리스 노트 입력
7. 배포
8. 링크 공유 또는 공개 노출

공개 테스트는 언제 좋은가:

- 앱이 거의 완성되어 있을 때
- 더 많은 기기와 사용 환경을 보고 싶을 때
- 출시 직전 베타 사용자 반응을 확인하고 싶을 때

공개 테스트는 언제 생략 가능한가:

- 비공개 테스트에서 충분한 사용자 검증을 마쳤을 때
- 빠르게 정식 출시할 계획일 때

---

## 8. 테스트 트랙 운영 시 주의사항

### 8-1. 내부 테스트 사용자와 다른 테스트 트랙

공식 문서 기준으로, 내부 테스트에 참여한 사용자는 바로 오픈/비공개 테스트를 받지 못할 수 있습니다.  
필요 시 내부 테스트를 opt-out한 뒤 다시 다른 테스트에 참여해야 합니다.

즉:

- 같은 사용자에게 내부 테스트와 비공개 테스트를 모두 할 경우
- 참여 링크와 상태를 혼동하지 않게 안내해야 합니다

### 8-2. 앱 검색 노출

- 내부 테스트: 검색 노출 안 됨
- 비공개 테스트: 일반 검색 노출 안 됨
- 공개 테스트: Google Play에 노출될 수 있음

### 8-3. 반영 시간

공식 문서 기준:

- 내부 테스트는 비교적 빠르게 반영
- 비공개/공개 테스트는 수 시간 지연될 수 있음

즉 테스트 링크를 보냈는데 바로 안 보인다고 해서 즉시 오류로 판단하면 안 됩니다.

---

## 9. 개인 개발자 계정이면 특히 중요한 점

Google 공식 문서 기준으로, `2023-11-13 이후 생성된 개인 개발자 계정`은 프로덕션 출시 전에 추가 테스트 요건을 충족해야 할 수 있습니다.

관련 문서:

- https://support.google.com/googleplay/android-developer/answer/14151465?hl=en

실무적으로 의미하는 바:

- 내부 테스트만 하고 바로 출시가 안 될 수 있음
- 비공개 테스트 정보와 테스트 진행 내역이 중요함
- Play Console이 요구하는 테스트 요건을 직접 확인해야 함

따라서 개인 계정이라면:

1. 내부 테스트
2. 비공개 테스트
3. 요구조건 충족 확인
4. 그 다음 프로덕션 출시

이 순서를 기본으로 보는 것이 안전합니다.

## 10. 프로덕션 출시 방법

경로:

- `Play Console > Test and release > Production`

진행 순서:

1. Production 페이지 진입
2. `Create new release`
3. 최신 `AAB` 업로드
4. 릴리스 노트 입력
5. `Review release`
6. 경고/누락 항목 해결
7. `Start rollout to production`

출시 전 반드시 완료되어 있어야 하는 것:

- 스토어 등록 정보
- 스크린샷, 아이콘, 대표 그래픽
- 개인정보처리방침 URL
- 광고 포함 여부
- Data safety
- 콘텐츠 등급
- 앱 액세스 정보
- 테스트 요구조건 충족 여부

## 11. 추천 실제 운영 순서

현재 `내차수첩` 기준 추천 순서는 아래와 같습니다.

### 1단계: 내부 테스트

목표:

- 앱 설치 가능
- 핵심 기능 동작
- 치명 버그 제거

대상:

- 본인
- 가까운 테스터
- QA 용도 사용자

### 2단계: 비공개 테스트

목표:

- 실제 사용 흐름 검증
- 여러 기기/여러 사용자 패턴 확인
- 데이터 안정성 확인

대상:

- 실제 차량 사용자
- 다차량 사용자
- 운행 빈도가 높은 사용자

### 3단계: 필요 시 공개 테스트

목표:

- 대규모 사용자 피드백
- 출시 전 공개 베타 검증

### 4단계: 프로덕션 출시

목표:

- 일반 사용자 대상 정식 공개

## 13. 내차수첩 기준 권장 결론

현재 앱은 아래 방식이 가장 현실적입니다.

1. 내부 테스트
2. 비공개 테스트
3. 문제 없으면 프로덕션 출시

공개 테스트는 선택 사항입니다.

즉 결론은:

- `내부 테스트`는 빠른 확인용
- `비공개 테스트`는 실제 출시 전 핵심 단계
- `공개 테스트`는 필요할 때만
- `프로덕션`은 최종 공개
